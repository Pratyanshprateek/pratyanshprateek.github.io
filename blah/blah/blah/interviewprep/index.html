PratStrike Project Interview Guide

Goal: Yeh page aapka full project cheat-code hai. Agar koi pooche “yeh game kaise banaya?”, “AI kaise kaam karta hai?”, “boss wave pe kya architecture tha?”, ya “C++ me game feel kaise achieve kiya?”, toh aap confidently answer kar sako.

One-Line Intro

PratStrike is a 2D top-down shooter built in C++ + SDL2, focused on gameplay systems, combat feel, enemy AI, boss phases, collision handling, menus, difficulty scaling, leaderboard persistence, and a complete playable loop.

1. High-Level Story: How We Built This From Scratch

Q: Project start ka thought process kya tha?
Sabse pehle idea simple tha: ek polished top-down shooter banana jo sirf “working game” na lage, balki “haan bhai, isme feel hai” lage. Isliye start se hi focus 3 cheezon par tha: architecture, combat systems, aur player feedback.

Humne pehle rendering aur engine shell banaya, phir ECS structure, phir movement/collision, phir bullets/enemies, phir AI, phir boss, phir wave progression, phir menus/leaderboard/pause.

Seedha boss se start karte toh boss bhi bolta “bhai pehle base toh bana lo.”
Q: Why C++ and SDL2?
C++ choose kiya because performance, memory awareness, and systems-level control chahiye tha. SDL2 choose kiya because it gives enough low-level control without forcing a big engine workflow. It handles windowing, input, rendering, audio, and fonts nicely, while the gameplay architecture stays fully ours.
Q: Final architecture kya bani?
Architecture roughly aise hai:

Engine handles app lifecycle and fixed-timestep loop.
GameScene is the orchestrator.
Registry/ECS stores entities + components.
Systems handle one job each: Input, Physics, Bullet, AI, Boss, Wave, Render, GameFeel helpers.

Isse coupling kam hota hai. Har system ka ek focused responsibility hai.

2. ECS Deep Explanation

Q: Why ECS instead of classic OOP inheritance like Player : Character : Entity?
ECS is better when behavior is composition-based. Example: player, enemy, bullet, pickup, boss, particle sab “different classes” nahi, balki component combinations hain.

Example:
Player = Transform + Velocity + Health + Sprite + PlayerState + Tag
Bullet = Transform + Velocity + Collider + Bullet + Sprite + Tag
Boss = Transform + Velocity + Health + Collider + Boss + Sprite + Tag

Inheritance me jaldi spaghetti ban sakti thi. ECS me reusable data blocks aur clean system logic milta hai.
Q: Registry ka role kya tha?
Registry entities banata/destroy karta hai aur component arrays manage karta hai. Hamara ECS sparse-set style component storage use karta hai, jisme component lookups efficient hain aur systems relevant component combinations par iterate kar sakte hain.
Q: Typical ECS flow ka code example?
Example: 
// entity create
std::uint32_t enemyId = registry.createEntity();

registry.addComponent(enemyId, TransformComponent{pos, pos, 0.0f, 0.0f});
registry.addComponent(enemyId, VelocityComponent{Vector2(), 0.0f, 120.0f, 180.0f, 0.04f, 200.0f});
registry.addComponent(enemyId, HealthComponent{60, 60, 0.0f, 0.12f, false});
registry.addComponent(enemyId, TagComponent{TagComponent::Tag::Enemy});
Yeh flexible hai, aur systems fir view<TransformComponent, VelocityComponent>() pe operate karte hain.

3. Game Loop and Timing

Q: Why fixed timestep use kiya?
Physics aur gameplay ko frame-rate independent banana tha. Agar variable timestep hota, toh different machines par dash timing, collision consistency, fire cooldown, hitstop duration, sab alag feel kar sakte the.

Isliye fixed update loop use kiya with something like: 
while (accumulator >= FIXED_DT) {
    onUpdate(FIXED_DT);
    accumulator -= FIXED_DT;
}
float alpha = accumulator / FIXED_DT;
onRender(alpha);
Yeh game logic ko stable rakhta hai aur rendering ko smooth banata hai.
Q: Render interpolation kya karti hai?
Physics fixed steps me chal rahi hoti hai, but render har frame hota hai. So we interpolate between previous and current position: 
Vector2 renderPos = prevPosition.lerp(position, alpha);
Isse visuals smoother lagte hain, especially movement aur camera me.

4. Input and Player Movement

Q: Player movement ka evolution kya tha?
Initially rotate-and-thrust style tha, but user feedback se clear hua ki top-down shooter me direct directional movement chahiye. So movement change karke proper WASD / Arrow key vector movement + mouse aim kiya.

Final behavior:
movement keys = position/movement vector
mouse = aim direction
left click/space = fire
shift = dash
Q: Dash ka implementation kaise kiya?
PlayerStateComponent me dash timer, cooldown, duration aur dash direction store kiya. Jab dash trigger hota hai: 
ps.isDashing = true;
ps.dashTimer = ps.dashDuration;
ps.dashCooldown = ps.dashCooldownMax;
ps.dashDirection = movementDir;
Aur update me velocity temporarily high set hoti hai.

5. Physics and Collision

Q: PhysicsSystem kya handle karta hai?
PhysicsSystem handles:
movement integration
friction
world bounds clamping
i-frame timer update
quadtree rebuild
collision detection
collision event generation
Q: Collision detection brute-force kyun nahi?
Bullets, enemies, pickups, player, boss sab ka pairwise check karna expensive ho sakta tha. Isliye Quadtree use kiya to spatially partition colliders and query nearby objects only.
Q: Boss wave pe crash kyun hua tha?
Yeh important interview point hai.

Boss collider large tha. Old quadtree insertion boundary-spanning objects ko child nodes me recursively daalta ja raha tha, jis se stack overflow due to excessive recursion hua. Crash report ne clearly dikhaya:
Thread stack size exceeded due to excessive recursion

Fix:
if object fully fits in exactly one child -> insert there
else -> keep it in current node

Translation: boss ko zabardasti chaar dabbo me ghusana band kiya.
Q: Collision response ka example?
Solid colliders overlap hon to separation by smallest axis: 
if (overlapX < overlapY) {
    // separate on X
} else {
    // separate on Y
}
Trigger colliders event create karte hain but physical push nahi karte.

6. Bullet System

Q: BulletSystem kya karta hai?
Bullet spawn, travel, max range cleanup, collision-based damage, score updates, and destruction. Bullets entities hain, koi special hardcoded projectile array nahi.
Q: Player bullet nozzle issue kaise fix kiya?
Initially bullets center ya sideways offset se spawn ho rahe the, isliye visually nozzle se nahi lag rahe the. Fix me player facing direction ke along forward muzzle offset use kiya: 
Vector2 forward = Vector2(0.0f, -1.0f).rotated(transform.rotation);
Vector2 muzzleOffset = forward * 28.0f;
Fir bullet spawn position = player position + muzzle offset.
Q: Enemy bullet damage register nahi ho raha tha, why?
Major reason tha enemy bullet hitbox too skinny feel ho raha tha, so visual contact aur collision contact mismatch tha. Fix me enemy bullet collider thoda enlarge kiya: 
const float colliderWidth = fromPlayer ? 6.0f : 12.0f;
const float colliderHeight = fromPlayer ? 12.0f : 20.0f;
Isse hit registration fairer feel hui.

7. Enemy AI

Q: AI state machine kya tha?
Enemies use a simple but effective state machine:
Patrol
Chase
Attack
Dead

Transitions distance-based aur hysteresis-based the to avoid jitter.
Q: Enemy types me difference kaise create kiya?
Har enemy type ke alag speed, radius, fire rate, damage, and behavior modifiers hain:

Grunt: basic pressure enemy
Flanker: player ke around offset path leta hai
Tank: slow but heavy and pushy
Sniper: long-range firing, stationary attack
Q: Difficulty-based aim logic kaise ki?
Difficulty ko pure HP sponge ya damage multiplier nahi banaya. Aim behavior and engagement style tune kiya.

Easy: wider aim error, less predictive shots
Medium: moderate correction
Hard: tighter aim but still intentionally imperfect

Example: 
float angleError = aimErrorForDifficulty(difficultyIndex, enemy.type);
float shotAngle = aimDirection.angle() + randomFloat(-angleError, angleError);
So bots human-like lagte hain, aimbot nahi.
Q: Enemies passive kyun lag rahe the and how did we fix it?
Initially enemies mostly close-range attack state me hi fire kar rahe the. Fix me chase state ke andar bhi ek pressure radius add kiya gaya. Matlab woh aapke upar chipakne se pehle hi fire pressure create karte hain.

This made combat feel more like an actual shooter and less like “aao bhai pehle hug karte hain phir goli chalate hain”.

8. Boss System

Q: Boss fight architecture kya thi?
BossSystem ek dedicated system tha. Boss is a normal entity with extra components: Transform, Velocity, Collider, Health, Boss, Sprite, Tag.
Q: Boss phases ka logic?
Health thresholds-based:
Phase 1: hp > 400
Phase 2: hp 201–400
Phase 3: hp <= 200

Phase transition pe: screenshake
hitstop
particles
explosion sound
movement and attack pattern escalation
Q: Boss attacks kya the?
Phase 1: spread shots
Phase 2: spiral burst
Phase 3: alternating spiral + charge

Goal yeh tha ki boss mechanically aur visually dono ways me escalate ho.

9. Wave System

Q: Wave progression kaise built hua?
WaveSystem pre-defined configs use karta hai for early waves, then procedural scaling for later waves. Wave 5 aur 10 boss waves hain.

Jab enemiesAlive 0 ho jata hai, then between-wave timer start hota hai, then next wave spawn hota hai.
Q: Wave 3 ke baad enemies not spawning issue kya tha?
Us type ke issues usually count logic, waiting flag, boss active state, ya dead entity cleanup timing se aate hain. We tightened the flow so dead enemies destroy hone ke baad wave count refresh hota hai and next wave reliably triggers.

10. Game Feel Systems

Q: “Game feel” exactly kaise implement kiya?
Yeh project ka heart tha. Implemented:
Hitstop
Screenshake
Hit flash
Death particles
Spatial audio

Yeh sab utility functions ke through trigger hote hain from collisions, enemy death, boss phase change, player hit, etc.
Q: Hitstop ka implementation kya tha?
Dedicated world hitstop entity par timer add karte hain. Physics and AI us timer ko respect karte hain. So when hitstop active:
physics freeze
AI freeze
render/UI still continue

Isse impact heavy lagta hai.
Q: Screenshake ka logic?
Camera entity me ScreenshakeComponent store hota hai with intensity, duration, timer, offset. Render update me decaying random offset generate hota hai.
Q: Spatial audio kaise kiya?
Sound source aur player ke world position ke basis par volume aur left/right panning calculate kiya. 
float pan = clamp((soundX - playerX) / maxRange, -1.0f, 1.0f);
Mix_SetPanning(channel, leftVol, rightVol);

11. Menus, Pause, Leaderboard

Q: Why add menu, difficulty, name entry, leaderboard?
Because “playable prototype” aur “presentable portfolio game” me farq hota hai. Game ko feel complete banana tha. So:
name entry
difficulty selection
start menu
pause menu
restart flow
persistent leaderboard
top score to beat
Q: Leaderboard persistence kaise ki?
Simple text file based persistence: 
name|score|wave|difficulty
On load: parse file, sort descending, keep top 3. On submit: append current score, sort, trim to 3, save back.
Q: Pause me score save kaise handle kiya?
Pause screen gives:
Resume
Save score and go to menu
Save score and exit

If run active hai and player exits from pause or closes app, current score submit hota hai before leaving.
Q: Esc app close bug kya tha?
Engine layer me hardcoded Esc => running = false tha. Is wajah se scene-level pause logic bypass ho raha tha. Fix: remove global escape-close behavior and route Esc to GameScene pause state.

13. Great Interview Answers Ready-to-Speak

Q: What was the hardest bug you solved?
The hardest bug was a boss-wave crash caused by excessive recursion in the quadtree. The boss had a large collider, and the previous insertion logic kept trying to push a boundary-spanning object deeper into child nodes. I used the macOS crash report to identify a stack overflow in Quadtree::insertInternal(), then changed the quadtree logic so objects that do not fully fit inside a single child remain in the current node. That fixed the crash and made the spatial partitioning more correct.
Q: What did you optimize for in this project?
I optimized for maintainable gameplay systems and good combat feel. Instead of just rendering sprites and moving entities, I focused on making the architecture scalable and the moment-to-moment feedback satisfying. That’s why I used ECS, fixed timestep updates, quadtree queries, and dedicated game-feel systems.
Q: If you had more time, what would you improve?
I would add a level editor, more enemy archetypes, better visual effects like GPU particles and trails, more animation states, and improved balancing tools. I would also add analytics or debug overlays for AI tuning and combat telemetry.

14. Advanced Interview Questions From This Project

Advanced Q1: Why is ECS a better fit here than deep inheritance?
Because entity behavior is composition-driven, not taxonomy-driven. Bullets, pickups, enemies, boss, particles all reuse overlapping data patterns. ECS reduces brittle inheritance chains and keeps systems data-oriented.
Advanced Q2: What problems does fixed timestep solve that variable timestep doesn’t?
Fixed timestep improves determinism, stabilizes physics/cooldowns/collision behavior, and makes balancing more consistent across different frame rates. Variable timestep can introduce inconsistent gameplay feel and tunneling-like issues.
Advanced Q3: Why was render interpolation still needed if physics already works?
Physics correctness and render smoothness are different concerns. Interpolation hides the visible stepping between fixed simulation updates and makes movement/camera visually smoother.
Advanced Q4: What is the tradeoff of quadtree vs brute-force collision?
Quadtree reduces average collision candidate count and scales better with many entities. Tradeoff is implementation complexity and edge cases like large objects spanning multiple regions.
Advanced Q5: How did you keep enemy difficulty fair instead of frustrating?
I scaled aim confidence and engagement behavior rather than just increasing raw damage or perfect tracking. Easy enemies miss more and predict less; hard enemies are more consistent but still intentionally imperfect.
Advanced Q6: How would you prevent bullet tunneling at higher projectile speeds?
I would use swept collision tests, raycast-style segment checks between previous and current positions, or sub-stepping for fast projectiles. Current AABB overlap is acceptable for this scale, but swept tests would be the next upgrade.
Advanced Q7: Why use hitstop as world-state instead of per-system hacks?
A central hitstop state gives one source of truth. Physics and AI can freeze consistently while render/UI continue, which is cleaner and easier to reason about than sprinkling pause flags everywhere.
Advanced Q8: How would you make this architecture multiplayer-friendly?
I would push further toward deterministic simulation, isolate side effects more cleanly, add input recording/rollback-safe structures, and rethink audio/render-only systems as presentation layers on top of deterministic game state.
Advanced Q9: What is one example of turning user feedback into an engineering change?
Player movement and enemy pressure were both reshaped through feedback. The project moved from rotate/thrust controls to direct top-down movement, and enemy AI was changed to engage from better distances with softer aim behavior to feel more like a normal shooter.
Advanced Q10: What did this project prove about you as an engineer?
It showed that I can design gameplay systems end-to-end, debug real runtime failures, iterate based on feel and feedback, and ship a complete interactive product rather than isolated demo features.
